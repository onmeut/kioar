All services run as Docker containers in a single Compose stack
(`docker-compose.prod.yml`). TLS is handled by Caddy (auto-ACME).
There is no Kubernetes, no CDN, no managed PaaS layer.

The Next.js process is a **long-running Node.js process**, not
a serverless/Lambda runtime. This distinction matters: the Postgres
connection pool (`globalThis.__kioarDb`) is a process-level singleton
intentionally. See `src/db/index.ts` for the warning about copying this
pattern to edge/serverless runtimes.

## 2. Database topology

| Item               | Value                                                                                                                                           |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| Engine             | PostgreSQL 16                                                                                                                                   |
| ORM                | Drizzle ORM (`drizzle-orm/postgres-js`)                                                                                                         |
| Schema             | `src/db/schema.ts` ‚Äî one file, ~40 tables                                                                                                       |
| Migrations         | `drizzle/` ‚Äî numbered SQL files, generated by `drizzle-kit generate`                                                                            |
| Connection pool    | `postgres` driver, `prepare: false` (pgbouncer-compatible), default 10 connections per process (`DATABASE_POOL_MAX`)                            |
| Timeouts           | `statement_timeout` 15 s, `idle_in_transaction_session_timeout` 10 s (server-side hard caps, not app-logic)                                     |
| Scaling path       | Add PgBouncer in transaction mode (Compose profile `pgbouncer`) when running >2 app instances                                                   |
| Slow query tracing | `pg_stat_statements` ‚Äî enable via `shared_preload_libraries` + `CREATE EXTENSION IF NOT EXISTS pg_stat_statements;`. No app-side wiring needed. |

### Data conventions

- **Timestamps**: `timestamp({ withTimezone: true })`, always stored as UTC.
  Never store Shamsi strings, timezone offsets, or abbreviations as truth.
- **Money**: `bigint` column `toman` (no fractional currency).
  VAT rate from `BILLING_VAT_RATE` env, never hardcoded.
- **Multi-tenancy**: pages are the owned entity (`profiles` table,
  `userId` = owner). A user can have many pages. Always resolve the current
  page with `resolveCurrentPageForOwner()` ‚Äî never assume one user = one page.

### Migration policy

1. `npm run db:generate` creates a new numbered migration from schema diff.
2. **Never hand-edit existing migration files.** Drizzle's meta journal
   checksums them; edits will break `db:migrate`.
3. `db:push` (dev only) applies schema changes without a migration file ‚Äî
   use for local iteration, commit the generated migration before merging.
4. Data backfills go in their own migration file, not the schema-change file
   that created the column.

## 3. Caching topology

### Public profile cache (Redis read-through)

The public profile page (`/[slug]`) is the highest-traffic route.
Its DB query is multi-table and expensive. It is wrapped in a Redis
read-through cache.

```
GET /[slug]
    ‚îÇ
    ‚îî‚îÄ‚ñ∫ getPublicProfileBySlug()        src/lib/data.ts
            ‚îÇ
            ‚îî‚îÄ‚ñ∫ withProfileCache(slug, loader)   src/lib/cache/profile-cache.ts
                    ‚îÇ
                    ‚îú‚îÄ Redis HIT ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚ñ∫ return cached payload
                    ‚îÇ   (key: kioar:page:v1:{slug}, TTL 300 s)
                    ‚îÇ
                    ‚îú‚îÄ Redis MISS ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚ñ∫ loadPublicProfileBySlug()
                    ‚îÇ                                 (DB multi-table query)
                    ‚îÇ                                 ‚îî‚îÄ‚ñ∫ write to Redis
                    ‚îÇ                                 ‚îî‚îÄ‚ñ∫ return fresh payload
                    ‚îÇ
                    ‚îî‚îÄ Redis ERROR (fail-open) ‚îÄ‚îÄ‚îÄ‚ñ∫ loadPublicProfileBySlug()
```

**404 sentinel**: a `null` return from the loader is stored as
`{"__kioarNotFound":true}` with a 60 s TTL. This limits repeated DB
hits on slug-spray attacks.

**Invalidation rule** (mandatory ‚Äî read before writing to the DB):
Any write path that affects what the public page renders must call one of
the `invalidateProfileCache*` helpers in `src/lib/cache/profile-cache.ts`
**after the transaction commits** (never inside a `tx`). The three helpers:

| Helper                                         | When to use                                                                 |
| ---------------------------------------------- | --------------------------------------------------------------------------- |
| `invalidateProfileCacheBySlug(slug)`           | Slug in scope (profile mutations, link edits, block changes, page deletion) |
| `invalidateProfileCacheById(pageId)`           | Only pageId in scope (billing, admin, cron, entitlement paths)              |
| `invalidateProfileCacheOnSlugChange(old, new)` | Slug rename ‚Äî drops both keys                                               |

**Metrics**: fire-and-forget Redis `INCR` on `kioar:metrics:profile_cache:{hit|miss|not_found_hit|error}`. No per-slug counters (unbounded cardinality).

**Health probe**: `GET /api/health` returns `{"ok":true,"db":"up","redis":"up|down|unconfigured","durationMs":N}`. Redis `"down"` does **not** set `ok: false` or return 503 ‚Äî Redis is a cache layer, not a data layer. A Redis outage degrades performance (every public page request hits the DB) but the app remains functional. Only DB down is a liveness failure.

### Rate limiting

Redis-backed sliding window buckets (`src/lib/rate-limit.ts`), with a
`rate_limit_buckets` Postgres table as fallback when Redis is unavailable.
Applied at all public mutation entrypoints (OTP, booking, form submit, signup).

## 4. Cron architecture

All scheduled work follows one pattern: `POST/GET /api/cron/<name>`,
guarded by `Authorization: Bearer $CRON_SECRET` + `pg_try_advisory_lock`.
External systemd timer (or Caddy scheduled jobs) hits the endpoint; the
endpoint self-guards against concurrent runs. No BullMQ, no pg_cron, no
Celery ‚Äî one pattern, one scheduler entry per job.

```
systemd timer
    ‚îÇ
    ‚îî‚îÄ‚ñ∫ POST /api/cron/<name>
            ‚îÇ
            ‚îú‚îÄ auth: timingSafeEqual(header, `Bearer $CRON_SECRET`)
            ‚îÇ
            ‚îú‚îÄ withRequestContext({ requestId, route: "cron.<name>" })
            ‚îÇ       ‚îî‚îÄ‚ñ∫ log.info("cron.<name>.start")
            ‚îÇ
            ‚îú‚îÄ pg_try_advisory_lock(<distinct BigInt key>)
            ‚îÇ       ‚îî‚îÄ locked=false ‚Üí log.info("cron.<name>.skipped") + 200
            ‚îÇ
            ‚îú‚îÄ work (startedAt = Date.now())
            ‚îÇ
            ‚îú‚îÄ log.info("cron.<name>.ok", { ...summary, durationMs })
            ‚îÇ
            ‚îî‚îÄ pg_advisory_unlock in `finally`
```

Current jobs:

| Route                        | Lock key           | Purpose                                                                 |
| ---------------------------- | ------------------ | ----------------------------------------------------------------------- |
| `/api/cron/cleanup`          | `7427301519462395` | Purge expired OTPs, sessions, rate-limit buckets                        |
| `/api/cron/billing`          | `4291884561927834` | Daily subscription state transitions (trial ‚Üí active ‚Üí grace ‚Üí expired) |
| `/api/cron/sms`              | `6128374910284756` | Drain the SMS queue via Kavenegar Lookup                                |
| `/api/cron/affiliate-unlock` | `3947561028473920` | Unlock matured affiliate commissions                                    |

### Tehran timezone in cron ‚Äî standing rule

All day-key generation in cron routes (and anywhere a `YYYY-MM-DD` key
anchors to a calendar day) **must use `tehranIsoDate(now)` from
`src/lib/date/persian.ts`**, never `now.toISOString().slice(0,10)`.

UTC midnight and Tehran midnight differ by 3.5 h (4.5 h in IRST, 3.5 h in
IRST, depending on DST). Using UTC causes day-buckets to shift at 8:30 or
9:30 pm local time instead of midnight, corrupting analytics aggregations
and billing period calculations for Iranian users. This bug was
fixed in Phase 2 of the pre-launch hardening pass and must not be
reintroduced.

Rule: **`tehranIsoDate(now)` for backend day-keys. Always.**

## 5. Billing source-of-truth

Billing state lives in five tables:

```
plans ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ features
  ‚îÇ                       ‚îÇ
  ‚îî‚îÄ‚îÄ‚îÄ plan_features ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò
           ‚îÇ
           ‚ñº
page_subscriptions   (current period, status, cancel-at-period-end)
           ‚îÇ
           ‚îî‚îÄ‚ñ∫ page_entitlements  (boolean or limit per lookup_key per page)
```

**Principle: no plan-name comparisons in product code.** The plan name is
display-only. All capability gating goes through:

```ts
pageHasFeature(pageId, "lookup_key"); // boolean gate
getPageEntitlementLimit(pageId, "lookup_key"); // numeric limit
```

Adding a new gateable capability requires running the Feature Registry
workflow documented in CLAUDE.md (propose key ‚Üí confirm plans ‚Üí seed ‚Üí
migrate ‚Üí wire both editor and renderer). Skipping any step creates an
inconsistency between what users can do and what they're billed for.

### Billing transitions

`applyVerifiedPayment` (`src/lib/billing-apply.ts`) is the single write
path for a successful payment: it updates `page_subscriptions`, rebuilds
entitlements in `page_entitlements`, and invalidates the Redis profile
cache. It logs `billing.apply.start` and `billing.apply.ok` with
pageId/planKey/durationMs for correlation.

Daily transitions (trial expiry, grace period expiry) are handled by
`/api/cron/billing` which calls `transitionForToday`.

## 6. Observability

### Logging

`src/lib/log.ts` ‚Äî structured JSON to stdout. Configured via `LOG_LEVEL`
env var (`debug | info | warn | error`, defaults to `info` in production,
`debug` in development). The emitter reads request context from
`AsyncLocalStorage` (`src/lib/log-context.ts`) and merges
`{requestId, route}` into every payload, so log lines emitted from
service files are automatically correlated to the request that triggered
them without passing a context argument.

Request-id injection pattern: every entry point (cron route handler,
API routes for user-facing flows) calls `withRequestContext(...)` at the
top. The `x-request-id` header is read from the incoming request and used
as the requestId if present (so Caddy-generated IDs propagate); a
`crypto.randomUUID()` is generated if the header is absent.

### Error monitoring

See `docs/error-monitoring.md` for the Sentry vs GlitchTip decision
(short answer: self-hosted GlitchTip, planned for first week post-launch).
No SDK is installed yet.

### Metrics

Redis `INCR` keys under `kioar:metrics:profile_cache:*` for cache
hit/miss/error ratios. Structured log fields (`durationMs`, `scanned`,
`applied`, etc.) in cron and billing logs provide timing and volume
visibility without a separate metrics pipeline.
